tiledb.cloud.soma.run_ingest_workflow¶

tiledb.cloud.soma.run_ingest_workflow(*, output_uri: str, input_uri: str, measurement_name: str, pattern: str | None = None, extra_tiledb_config: Dict[str, object] | None = None, platform_config: Dict[str, object] | None = None, ingest_mode: str = 'write', resources: Dict[str, object] | None = None, namespace: str | None = None, access_credentials_name: str | None = None) → Dict[str, str]¶

Starts a workflow to ingest H5AD data into SOMA.

Parameters:

  • output_uri – The output URI to write to. This will probably look like tiledb://namespace/some://storage/uri.

  • input_uri – The URI of the H5AD file(s) to read from. These are read using TileDB VFS, so any path supported (and accessible) will work. If the input_uri passes vfs.is_file, it’s ingested. If the input_uri passes vfs.is_dir, then all first-level entries are ingested . In the latter, directory case, an input file is skipped if pattern is provided and doesn’t match the input file. As well, in the directory case, each entry’s basename is appended to the output_uri to form the entry’s output URI. For example, if a.h5ad` and ``b.h5ad are present within input_uri of s3://bucket/h5ads/ and output_uri is tiledb://namespace/s3://bucket/somas, then tiledb://namespace/s3://bucket/somas/a and tiledb://namespace/s3://bucket/somas/b are written.

  • measurement_name – The name of the Measurement within the Experiment to store the data.

  • pattern – As described for input_uri.

  • extra_tiledb_config – Extra configuration for TileDB.

  • platform_config – The SOMA platform_config value to pass in, if any.

  • ingest_mode – One of the ingest modes supported by tiledbsoma.io.read_h5ad.

  • resources – A specification for the amount of resources to provide to the UDF executing the ingestion process, to override the default.

  • namespace – An alternate namespace to run the ingestion process under.

  • access_credentials_name – If provided, the name of the credentials to pass to the executing UDF.

Returns:

A dictionary of {"status": "started", "graph_id": ...}, with the UUID of the graph on the server side, which can be used to manage execution and monitor progress.
